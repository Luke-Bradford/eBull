# #1265 â€” jobs-process lazy master-key reload when creds added post-startup

## Problem (ETL readiness audit, confirmed completion blocker #2)
Default deployment ships `EBULL_SECRETS_KEY` empty (file-secret path). On a fresh DB the jobs process boots with `install_key=False` (`master_key.bootstrap`, master_key.py:428 â€” `state=clean_install`, no env key) â†’ the process-global AES-GCM key (`secrets_crypto._aesgcm`) stays `None`. The operator then adds broker creds via the **API** process; its first-save lazy-gen creates the root-secret file and encrypts the cred â€” but the **jobs** process global is never reloaded (sole `bootstrap` call is at jobs boot, app/jobs/__main__.py:382). When the bootstrap orchestrator runs S1 `universe_sync`, `_load_etoro_credentials` â†’ `load_credential_for_provider_use` â†’ `secrets_crypto.decrypt` â†’ `_get_aesgcm()` raises `MasterKeyNotLoadedError`, which scheduler.py:1549 does **not** catch â†’ S1 errors â†’ `universe_seeded` cap dies â†’ full bootstrap cascade (~0.3s, "all stages instant-error").

Run-workaround (shipped in #1265 comment): set `EBULL_SECRETS_KEY` â†’ `install_key=True` unconditionally. This proposal is the **durable** fix for operators on the default file-secret path.

## Key facts (verified in code) â€” corrected after Codex ckpt-1 round 1
- **`bootstrap()` is NOT a safe hot-path "ensure" primitive.** It runs `_revoke_stale_ciphertext()` (master_key.py:401) which *mutates* â€” soft-revokes `no_key`/`mismatch` rows (master_key.py:290/293). Re-calling it per credential-using job would revoke creds on the hot path. The lazy reload MUST be read-only.
- **`bootstrap()`'s env-mismatch fail-loud is order-fragile**: it revokes mismatching rows *before* the `creds_exist` check, so if ALL active rows mismatch they get revoked â†’ `creds_exist` false â†’ the `MasterKeyError` branch is skipped (master_key.py:403â†’411). A read-only loader that checks mismatch on the *true* (un-revoked) cred set is strictly MORE faithful to ADR-0003 Â§9.
- `master_key.bootstrap` is lock-free (no advisory lock); it derives the key from `EBULL_SECRETS_KEY` (env) else the root-secret file. The API persists the root-secret file BEFORE the DB cred row (broker_credentials.py:243â†’248), so a jobs process that sees a cred row is guaranteed the file exists (no file-before-row race on the intended path).
- `secrets_crypto.set_active_key(key)` installs the AES-GCM under `_aesgcm_lock` (idempotent, thread-safe). No public "is loaded?" predicate exists yet.
- **No key is ever generated by this change** â€” lazy-gen stays the API first-save path; the jobs process only READS the existing env/file key.

## Design (smallest surface; read-only lazy load; single chokepoint)
1. `secrets_crypto.is_active_key_loaded() -> bool` â€” reads `_aesgcm is not None` under `_aesgcm_lock`. New predicate; no behaviour change.
2. `master_key.load_existing_broker_key(conn) -> bytes | None` â€” **READ-ONLY** key resolver built from the SAME primitives `bootstrap` uses (`decode_env_key` / `read_root_secret` + `derive_broker_encryption_key`, `_credentials_exist`, `_newest_active_decryptable`) but with **NO `_revoke_stale_ciphertext` and NO state mutation**:
   - env key set â†’ `derive = decode_env_key(...)`; else root secret present â†’ `derive = derive_broker_encryption_key(read_root_secret())`; else â†’ return `None`.
   - **ADR-0003 Â§9 fail-loud (preserved + corrected):** if env key set AND `_credentials_exist(conn)` (true count, no prior revoke) AND not `_newest_active_decryptable(conn, derive)` â†’ `raise MasterKeyError` (env key set but doesn't match the NEWEST active ciphertext = misconfig; identical check + fidelity to `bootstrap`, ADR-0003 Â§9). The file path does NOT raise on undecryptable (file-derived key is authoritative; a bad match surfaces as `CredentialDecryptError` at use time â€” AES-GCM-safe).
   - return `derive` (or `None`).
3. `master_key.ensure_broker_key_loaded(conn) -> bool`:
   - if `is_active_key_loaded()` â†’ return True (steady-state no-op; no DB touch).
   - else `k = load_existing_broker_key(conn)`; if `k is not None` â†’ `set_active_key(k)`, log INFO "lazily loaded broker key (#1265)", return True; else return False.
   - Read-only + idempotent `set_active_key` â†’ re-entrancy-safe across concurrent jobs without a lock (worst case: two reads + two idempotent installs of the same key). A `MasterKeyError` from the env-mismatch guard propagates (fail-loud, not swallowed).
4. `scheduler._load_etoro_credentials`: after opening `conn`, before the credential loads, call `ensure_broker_key_loaded(conn)`. **No new `except` clause.** If the key is genuinely underivable, the subsequent decrypt still raises `MasterKeyNotLoadedError` and it **propagates as a loud stage error** â€” NOT converted to a clean `return None`. Rationale (Codex BLOCKING): the bootstrap stage wrapper treats a normal return as stage *success*, and S1 provides the `universe_seeded` cap on success â€” converting a master-key failure to `return None` would let the bootstrap "complete" with no universe (the silent-empty catastrophe). A genuine no-key MUST fail loud so the operator fixes it. The happy path (#1265: creds added, file present) loads the key and `universe_sync` runs normally.

## Why this chokepoint
`_load_etoro_credentials` is the shared credential-load helper for BOTH the bootstrap-direct invoke path (the failing one) and steady-state scheduled jobs, so one fix covers all callers. The executor's `_credential_health_blocks` gate is executor-only and bootstrap bypasses it (audit finding) â€” fixing here is the correct single point.

## Tests (synthetic; DB-backed where needed)
- `is_active_key_loaded` reflects set/clear.
- `load_existing_broker_key`: (a) env key set â†’ returns decoded key, no DB mutation (assert no revoke side-effect on a seeded mismatch-free row); (b) no env key + root secret present â†’ returns file-derived key; (c) neither â†’ `None`; (d) **fail-loud**: env key set + existing cred ciphertext not decryptable under it â†’ raises `MasterKeyError`; (e) **no-revoke proof**: a `mismatch`/`no_key` row present is NOT soft-revoked by a call (status unchanged) â€” the key behavioural difference from `bootstrap`.
- `ensure_broker_key_loaded`: already-loaded â†’ True + no DB touch; unloaded + derivable â†’ installs + True; unloaded + not derivable â†’ False, stays unloaded.
- `_load_etoro_credentials` (#1265 end-to-end): key cleared + creds present + key derivable â†’ lazy reload â†’ returns creds; key cleared + NOT derivable â†’ `MasterKeyNotLoadedError` propagates (NOT swallowed, NOT `None`) â†’ asserts the stage would error, not silently succeed.

## Out of scope
- Cross-process push notification of "creds added" (the lazy pull is self-healing â€” the first credential-loading job reloads).
- **Stale loaded-key self-heal** (Codex IMPORTANT): if the jobs process already has an OLD key cached and the root secret is rotated/restored under a new key, the loaded-bit guard short-circuits and decrypt fails with `CredentialDecryptError` (AES-GCM prevents wrong plaintext â€” no data corruption). Key rotation is a distinct operator flow (ADR-0001 manual rotation + process restart); not solved here. Documented, not silently ignored.

## Security review (ADR-0001 / ADR-0003)
- No silent key generation â€” jobs only reads the env/file key. âś“
- ADR-0003 Â§9 env-mismatch fail-loud preserved (and corrected vs bootstrap's revoke-first ordering): read-only loader raises on env-set + un-decryptable NEWEST active ciphertext (same single-row check `bootstrap` uses â€” matched fidelity, not a new all-active scan). âś“
- No mutation on the credential-load hot path (read-only loader; the boot-time `bootstrap` revoke pass is untouched). âś“
- No decrypt without a key; a genuine no-key stays a loud stage error (never silent success / silent-empty universe). âś“
- Auditable: lazy reload logs INFO; credential access + revoke audit rows unchanged.

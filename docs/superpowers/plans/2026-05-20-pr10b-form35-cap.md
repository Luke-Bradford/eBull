# PR10b — Form 3 / Form 5 latest-per-pair cap

> Tracking: **#1233** PR10b. Spec §4.4 + §7 + §12.
>
> Split from PR10 because Form 3/5 are observations-shaped at write,
> while business summary (PR10a, #1246, merged 2026-05-20) was
> structurally latest-only via `instrument_business_summary.instrument_id`
> PK. PR10b therefore needs a different shape per source.

## 1. Scope

Two sources, each with its own load-bearing chokepoint surface:

| Source | Cap shape | Why this shape |
| --- | --- | --- |
| **Form 3** | Read-side latest-per-pair via `DISTINCT ON` (already wired); PR10b PINS the SELECT shape AND the Form-4-exclusion anti-join via lint guard + adds a multi-pair behaviour test | `insider_filings.accession_number` is PK; appending one row per accession is the schema contract. Form 3 lifetime volume per issuer is bounded (~5-30 lifetime filings), and a 10y-old Form 3 for a still-serving officer IS the correct baseline — `INSIDER_FORM3_BACKFILL_FLOOR_YEARS = None` stays. The read-side `DISTINCT ON (filer_cik, security_title, is_derivative) ORDER BY as_of_date DESC, accession_number DESC, row_num ASC` + the `NOT EXISTS` anti-join against non-tombstoned Form 4 rows is the full operator-visible invariant for `list_baseline_only_insider_holdings` ([insider_form3_ingest.py:778-836](app/services/insider_form3_ingest.py#L778-L836)). PR10b's job: pin both halves against silent drift. |
| **Form 5** | Ingest-time `filed_at` horizon helper (**18 months**) at every writer chokepoint | Form 5 transactions land in `insider_transactions` (PK `accession_number` + per-row `txn_row_num`) just like Form 4. Latest-per-pair at write time is not enforceable without a schema overhaul. An 18-month `filed_at` cap is the closest annual-cadence approximation with the annual-cadence buffer Codex 1a flagged: a 1y cap drops the only/latest Form 5 for a pair for weeks (or indefinitely if no newer annual is required). 18m closes that gap while keeping volume tight — Form 5 cadence is ~annual, so at most 1-2 admitted per pair. Spec §4.4 + handover §12 noted 1y as the proposed default with audit; the audit pushes to 18m. |

Out of scope: any schema change (PR10b is read-side pin + parser gate +
sync gate only); no `DELETE FROM` against pre-cap rows (§6.3); no Form
3 horizon cap (would shed correct baselines for long-serving officers);
no Form 4 sync_insiders gate (separate PR4 follow-up — see §7 risk
log).

## 2. Empirical Form 5 audit (2026-05-20, dev DB)

Form 5 volume on dev DB (`filing_events.filing_type IN ('5','5/A')`):

```
total events  : 45,845     spanning 1996-02-20 → 2026-04-28
by filing year: 2026 → 747
                2025 → 878
                2024 → 893
                2023 → 1,590
                2022 → 1,838
                2021 → 1,529
                2020 → 1,750
                2019 → 1,645
                2018 → 1,983
                2017 → 1,750
```

`insider_filings.document_type='5'` is 0 — the manifest worker hasn't
drained any Form 5 yet, and the bulk dataset ingest writes to
observations only (no `insider_filings` row from bulk path).

Volume per filer per year ≈ 750-1900 / 5,174 US filers ≈ 0.15-0.40 —
the long tail of insiders who occasionally need Form 5 catch-up.
An 18-month `filed_at` window therefore admits ≤2 Form 5s per
insider-issuer pair on the typical case (one at year boundary, one
mid-window after the next annual filing lands), and the most-recent
filing is never silently dropped while waiting for the next annual.

**Edge case**: an 18-month cap can briefly admit BOTH the prior-year
Form 5 and the current-year Form 5 for the same insider when both
fall inside the window. That's the deliberate trade-off for closing
the 1y annual-cadence gap. Operator-visible impact: zero — the read
side already de-dupes via cumulative Form 4 rollup
(`refresh_insiders_current` aggregates from observations + Form 3
baseline). Documented in the helper docstring.

## 3. Plan structure (mirrors PR4 + PR9)

### 3.1 Form 5 — Files touched

1. **`app/services/insider_transactions.py`** — add
   `INSIDER_FORM5_RETENTION_MONTHS = 18`,
   `form5_retention_cutoff(now=None)`,
   `form5_within_retention(filing_date, now=None)` helpers next to
   `form4_*`. Month arithmetic via `relativedelta`-style manual
   subtraction (stdlib only — Python `dateutil` is already a runtime
   dependency; check before adding). Boundary inclusive. Leap-day
   handling: Feb 29 → Feb 28 of the 18-months-earlier month.
2. **`app/services/manifest_parsers/insider_345.py::_parse_form5`** —
   add pre-fetch gate mirroring the Form 4 gate at line 162. Pre-cap
   rows tombstone with `error="retention floor"` and
   `document_type='5'`. No `filing_raw_documents` row is written —
   future cap widening recovers via `sec_rebuild`.
3. **`app/services/sec_insider_dataset_ingest.py`** — extend the
   existing Form-4-only retention block at lines 334-340 + 401-407 to
   also gate Form 5. Two new locals at the top of the loop:
   ```python
   retention_cutoff = form4_retention_cutoff()
   retention_cutoff_form5 = form5_retention_cutoff()
   ```
   Each per-row block computes its own predicate against the correct
   cutoff via a `if-elif` ladder:
   ```python
   form_upper = (form or "").strip().upper()
   if form_upper.startswith("4") and filed_at.date() < retention_cutoff:
       result.rows_skipped_retention += 1
       continue
   if form_upper.startswith("5") and filed_at.date() < retention_cutoff_form5:
       result.rows_skipped_retention += 1
       continue
   ```
   Replaces the PR4 carve-out comment ("Form 5 rows are NOT gated
   here — they ride to PR10's latest-only cap"). Two predicates per
   loop (`NONDERIV_TRANS` + `NONDERIV_HOLDING`).
4. **`app/services/ownership_observations_sync.py::sync_insiders`** —
   add Form 5 retention gate. `sync_insiders` reads from
   `insider_transactions` JOINed to `insider_filings` (no
   `filing_date` column on either) — extend with a **LEFT JOIN** to
   `filing_events fe ON fe.provider_filing_id = it.accession_number
   AND fe.provider = 'sec'` (the canonical filing_date source) AND
   filter:

   ```sql
   AND (
       f.document_type IN ('3','3/A','4','4/A')  -- Form 4 cap deferred (#PR4-followup)
       OR (
           f.document_type IN ('5','5/A')
           AND fe.filing_date IS NOT NULL
           AND fe.filing_date >= %(form5_cutoff)s
       )
   )
   ```

   `LEFT JOIN` (Codex 1b MED) so Form 4 rows lacking a
   `filing_events` row (legacy ingest paths that wrote
   `insider_transactions` without a manifest entry) still sync — the
   Form 4 branch is `document_type IN ('3','3/A','4','4/A')` and
   never references `fe.filing_date`. Form 5 branch requires the
   join row to exist (`fe.filing_date IS NOT NULL`) so a Form 5
   without a manifest entry is excluded — safe because every Form 5
   in steady state arrives via the manifest worker. Form 4
   explicitly carved out and tagged with a follow-up ticket so
   PR10b doesn't silently fix PR4's gap. Form 5 closed here.
5. **No `_ingest_single_accession` path for Form 5** — the per-filing
   path under `insider_transactions.py::_process_candidates` filters
   `filing_type IN ('4', '4/A')` (already excludes Form 5). Form 5
   only enters via the manifest worker + the bulk dataset + sync.
6. **No rewash, no SQL repair sweep, no `_apply_form5`** — Form 5
   shares Form 4's parser (`parse_form_5_xml` returns same shape) but
   has no rescue / repair path of its own.

### 3.3 Form 5 — Lint guard

`scripts/check_form5_retention.sh` — four placement invariants:

- **A.** `insider_transactions.py` defines `form5_retention_cutoff(`
  AND `form5_within_retention(` AND `INSIDER_FORM5_RETENTION_MONTHS`
  (helpers + constant in canonical module — exactly 1 def line each
  + exactly 1 constant assignment).
- **B.** `manifest_parsers/insider_345.py::_parse_form5` body
  contains `form5_within_retention(` AT LEAST ONCE (pre-fetch gate).
  Awk function-scope counter mirrors PR4's gate.
- **C.** `sec_insider_dataset_ingest.py` calls `form5_retention_cutoff(`
  exactly once (archive-level anchor) AND contains EXACTLY 2
  occurrences of the literal `filed_at.date() < retention_cutoff_form5`
  (one per write loop — NONDERIV_TRANS + NONDERIV_HOLDING). Mirrors
  PR4's `filed_at.date() < retention_cutoff` parity check.
- **D.** `ownership_observations_sync.py::sync_insiders` body
  contains `form5_retention_cutoff(` AT LEAST ONCE AND each of these
  literals appears AT LEAST ONCE: `LEFT JOIN filing_events fe`,
  `f.document_type IN ('5','5/A')`,
  `fe.filing_date >= %(form5_cutoff)s`. Codex 1b MED — pinning all
  three together so a future refactor cannot drop the LEFT
  semantics (would silently shed Form 4 rows missing a manifest
  entry) nor the cutoff predicate (would silently admit pre-cap
  Form 5).

Lint guard wired into `.githooks/pre-push` after
`check_business_summary_latest_only.sh`. Awk-based scope parsing
(BSD/GNU `grep -P` portability — PR4 Codex 1c lesson). Empty-grep

`docs/review-prevention-log.md:1460`).

### 3.4 Form 3 — Files touched

1. **No production code change for Form 3.** The DISTINCT ON shape +
   the anti-join at `app/services/insider_form3_ingest.py:778-836`
   (inside `list_baseline_only_insider_holdings`) are already the
   latest-per-pair contract. PR10b's contribution is pinning both
   against drift.
2. **`scripts/check_form3_latest_per_pair.sh`** — lint guard.
   Awk-based SQL-block accumulator that:
   1. Locates the `def list_baseline_only_insider_holdings(` line in
      `insider_form3_ingest.py`.
   2. Walks forward until the NEXT top-level `def ` / `class ` or
      EOF. Within that scope:
   3. Concatenates every non-blank line into a single normalised
      string with all internal whitespace collapsed to single
      spaces. Comments stripped via `gsub(/--[^\n]*/, "")` per line
      BEFORE concatenation (avoids SQL line-comments influencing the
      normalised string).
   4. Asserts (as literal-substring on the normalised string):
      - **A.** Exactly 1 `SELECT DISTINCT ON (` opener.
      - **B.** Exactly 1 occurrence of
        `iih.filer_cik, iih.security_title, iih.is_derivative`
        (after whitespace normalisation).
      - **C.** Each of `iih.as_of_date DESC,`,
        `iih.accession_number DESC,`, `iih.row_num ASC` appears
        exactly 1 time (tie-break in priority order).
      - **D.** The anti-join shape exists exactly 1 time:
        `WHERE NOT EXISTS ( SELECT 1 FROM insider_transactions it
        INNER JOIN insider_filings ft ON ft.accession_number =
        it.accession_number AND ft.is_tombstone = FALSE WHERE
        it.instrument_id = %s AND it.filer_cik = b.filer_cik )`
        (whitespace-normalised). Pins the Form-4-activity exclusion
        — Codex 1a MED.
3. **No write-path lint** for Form 3 — `insider_filings.accession_number`
   PK is the schema contract; per-accession append is correct.

## 7. Risk / open questions

- **PR4 sync_insiders Form 4 gap (deferred)** — Codex 1a flagged that
  `sync_insiders` does not gate Form 4 by retention. PR10b explicitly
  carves out Form 4 in the new sync WHERE clause and files a PR4
  follow-up ticket so the gap is tracked. Rationale: closing it now
  is scope creep into PR4's territory; the deferred ticket is the
  honest record.
- **18m vs 1y Form 5 cap** — initial plan proposed 1y; Codex 1a MED
  said 13m minimum, recommended 18m. This revision adopts 18m and
  documents the annual-cadence buffer. Helper constant
  `INSIDER_FORM5_RETENTION_MONTHS = 18` is the single tunable.
- **Form 5 tombstone direction** — pre-fetch tombstoning means no
  `filing_raw_documents` row. Recovery via `sec_rebuild` source-reset
  (same as Form 4 PR4). No new code path needed.
- **Form 3 DISTINCT ON pin via lint vs schema** — a schema PK on
  `(filer_cik, security_title, is_derivative)` against
  `insider_initial_holdings` would be a stronger guarantee but would
  break the existing append-one-row-per-accession contract (multiple
  3/A amendments for the same pair would conflict). Lint guard +
  whitespace-normalised SQL accumulator is the right grain.
- **Lint guard SQL-block accumulator robustness** — Codex 1a HIGH on
  v1's literal `iih.filer_cik, ...` match. v2 uses awk multi-line
  accumulator + whitespace collapse + SQL-comment strip. Resilient
  against future reformatting; brittle only against semantic changes
  (which is the desired behaviour).

# thesis-critic

## When to use

Any change to the critic path inside `app/services/thesis.py` — `_CRITIC_SYSTEM`
(line 607), `_build_critic_prompt` (line 639), `_call_critic` (line 717),
`_validate_critic_output` (line 744), the `_VALID_VERDICTS` frozenset (line 64), or
the critic token budget (`_MAX_TOKENS_CRITIC`, line 78). Also read it before touching
the `theses.critic_json` column (`sql/006_theses_critic_json.sql:4`) or how the thesis
endpoints surface it (`app/api/theses.py`). The writer half, the `theses` table, the
scoring/portfolio consumption of thesis fields, and the refresh jobs belong to the
neighbouring `thesis-writer` skill — the critic is one lens **inside** that engine, not
a separate service.

## What it is

The adversarial half of build priority #4 (`.claude/CLAUDE.md`). Every
`generate_thesis(...)` run (thesis.py) calls the Claude **writer** for the memo, then
`_call_critic(client, memo_markdown, context)` for a structured counter-thesis. Model
`claude-sonnet-4-6` (`_MODEL`, line 76), `max_tokens = 1024` (`_MAX_TOKENS_CRITIC`).
There is **no dedicated critic service, table, job, or endpoint** — it is a prompt +
validator + one JSONB column.

Role (`_CRITIC_SYSTEM`): a contrarian analyst that **attacks the current long thesis and
surfaces the strongest failure case** — fight confirmation bias, do not restate the bull
case, prefer the strongest realistic objection over generic cautions, cite specific
metrics/dates/events. It is explicitly told to mine the `risk_metrics` evidence block
(the `risk_v1` scalars assembled into `context`, thesis.py:499) for realized-risk
objections — deep drawdown, high beta, weak Calmar, fat-tail `var_5` — that the memo
glossed over.

`_validate_critic_output` requires exactly six fields: `summary`, `key_risks`,
`hidden_assumptions`, `evidence_gaps`, `thesis_breakers`, `verdict`. `verdict` must be
one of `_VALID_VERDICTS = {"Strong challenge", "Moderate challenge", "Weak challenge"}`
— no ad-hoc strings.

Storage: the validated dict lands in `theses.critic_json JSONB`
(`sql/006_theses_critic_json.sql`) via `_insert_thesis_atomic` — passed only when
truthy, else `NULL`. Never appended into `memo_markdown`.

Read: exposed as `ThesisDetail.critic_json` on `GET /theses/{instrument_id}` and
`GET /theses/{instrument_id}/history` (`app/api/theses.py`, `_THESIS_COLUMNS` selects
`t.critic_json`); regenerated by `POST /instruments/{symbol}/thesis`. No dedicated
frontend panel renders the critic fields today.

## Invariants

- **Critic output stored separately** (settled-decisions "Thesis semantics" → "Critic
  output"): critic text lives in `critic_json` only — never appended into
  `memo_markdown`.
- **Critic runs on every generation** (settled-decisions "Thesis semantics" → "Critic
  invocation"): the critic call fires for every thesis generation in v1. It is not gated
  or sampled.
- **Best-effort, never blocking**: `_call_critic` catches every exception and returns
  `{}` (logged); a critic failure stores the thesis **without** `critic_json` and never
  wedges a valid memo. Contrast the writer, whose invalid output raises and blocks the
  insert.
- **Verdict enum is closed**: only the three `_VALID_VERDICTS` strings pass validation —
  add a value here and in the prompt together, never one without the other.
- **Auditability / deterministic execution** (repo non-negotiables): the critic is
  research-layer evidence — it feeds no score and no execution path directly; it is the
  persisted contrarian audit trail beside every long-only, no-leverage memo. It must
  never soften into a rubber stamp.

## Failure conditions

Missing critical source data, stale timestamps beyond threshold, or contradictory
evidence must surface as **explicit signals — never a neutral default**.

- **Missing data stays a gap**: absent inputs belong in `evidence_gaps`, not silently
  assumed away. The prompt forbids treating a `benchmark_missing` beta as 0 and forbids
  citing a non-`ok` risk metric as a precise number — status flags pass through, and the
  critic cites `{window_key, as_of_date, metric_version}` rather than fabricating.
- **Contradiction is the product**: the critic exists to surface the strongest realistic
  objection and emits an explicit `verdict` (Strong/Moderate/Weak challenge) — a
  contradiction between memo and evidence is reported, not reconciled into a bland
  neutral note.
- **Critic silence is honest, not fake-green**: when the model fails or returns
  unparseable/schema-invalid JSON, the row is stored with `critic_json = NULL` and a
  warning logged — a real absence, never a fabricated empty-but-passing critique.
